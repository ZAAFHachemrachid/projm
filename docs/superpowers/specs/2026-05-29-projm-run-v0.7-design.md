# `projm run` + `pn` — v0.7 Design Spec

**Date:** 2026-05-29
**Status:** Draft
**Version:** 0.7

---

## 1. Overview

Add a `projm run` subcommand that auto-detects the project stack, predicts the dev command, and runs it directly — no need to leave the terminal or open an editor. Introduce a companion shell function `pn` that pairs a fuzzy project picker (like `pg`) with the run action.

---

## 2. Shell function: `pn`

### 2.1 Behaviour

Generated by `projm init` alongside `pg`. Same project-selection logic as `pg`, different action after selection.

```
pn              → fuzzy picker → run selected project's dev command
pn .            → run dev command for current directory
pn /path/to/dir → run dev command for that path
pn <query>      → like pg, pre-filtered picker or direct match → run
```

### 2.2 Shell functions

**zsh:**
```zsh
pn() {
    projm run "$@"
}
```

**PowerShell:**
```powershell
function pn {
  projm run $args
}
```

`projm run` handles process spawning internally — it inherits the terminal's stdin/stdout/stderr directly. There's no `eval` pattern like `pg` because the command runs in a child process, not a subshell. The shell function is needed so that `pn .` passes the current shell's CWD context to `projm run` and so the child process inherits the proper terminal for Ctrl+C and interactive input.

### 2.3 `projm init` integration

- `projm init` writes both `pg` and `pn` shell functions in the same projm block.
- The `--alias` flag controls `pg`'s name only; `pn` is always `pn`.
- Idempotent — re-running `init` updates both functions.

## 3. CLI subcommand: `projm run`

### 3.1 Signature

```
projm run [path-or-query]
```

### 3.2 Resolution order

**If arg is a path** (`.`, absolute, or relative):
→ Resolve dev command for that path and run it.

**If no arg:**
1. Check if CWD is inside a recognised project (via base directory scan or stack markers) → run it directly.
2. Otherwise → show interactive picker.

**If arg is a name** (not a path):
1. Single exact match (case-insensitive) in scanned projects → run directly.
2. Multiple matches → show picker pre-filtered by the query.
3. No matches → show full picker.

### 3.3 Output model

All `projm` UI (picker, status messages) goes to **stderr**. The child process inherits **stdout/stderr** directly — user sees the dev command output in real time. The shell function `pn` uses `2>/dev/tty` for UI and lets `projm run`'s stdout flow through naturally.

Exit code is forwarded from the child process.

### 3.4 Stdin handling

The child process inherits stdin so the user can interact with it (e.g. `cargo run` prompts, `bun dev` with HMR keybindings, `Ctrl+C` to stop).

## 4. Dev command prediction

### 4.1 Stack-to-command mapping

| Detection Marker                  | Category Hint       | Resolved Command                       |
| --------------------------------- | ------------------- | -------------------------------------- |
| `Cargo.toml` (no `package.json`)  | Rust binary/lib     | `cargo run`                            |
| `package.json` + `bun.lock`       | JS/TS (Bun)         | `bun run dev`                          |
| `package.json` + `pnpm-lock.yaml` | JS/TS (pnpm)        | `pnpm dev`                             |
| `package.json` + `yarn.lock`      | JS/TS (Yarn)        | `yarn dev`                             |
| `package.json` + `package-lock.json` | JS/TS (npm)      | `npm run dev`                          |
| `package.json` only (no lockfile) | JS/TS (unknown)     | `npm run dev`                          |
| `src-tauri/`                      | Tauri desktop app   | `<pkg-mgr> tauri dev`                  |
| `pubspec.yaml`                    | Flutter/Dart        | `flutter run`                          |
| `go.mod`                          | Go                  | `go run .`                             |
| `pyproject.toml` + `uv.lock`      | Python (UV)         | `uv run python main.py`                |
| `setup.py` / `requirements.txt`   | Python (pip)        | `python main.py` or `python3 main.py`  |
| `Gemfile` + `config/routes.rb`    | Ruby on Rails       | `bin/rails server`                     |
| `mix.exs`                         | Elixir/Phoenix      | `mix run --no-halt`                    |
| `build.gradle`                    | Kotlin/Java         | `./gradlew run`                        |
| `pom.xml`                         | Java/Maven          | `mvn compile exec:java`                |
| `composer.json` + `artisan`       | Laravel/PHP         | `php artisan serve`                    |
| `CMakeLists.txt`                  | C/C++               | `cmake --build build`                  |
| `*.csproj` / `*.sln`             | C#/.NET             | `dotnet run`                           |

### 4.2 `package.json` script resolution

When the stack uses `package.json`, resolve dev command in this priority:

1. **Presence of `bun.lock`** → `bun run <script>`
2. **Presence of `pnpm-lock.yaml`** → `pnpm <script>`
3. **Presence of `yarn.lock`** → `yarn <script>`
4. **Fallback** → `npm run <script>`

The script name resolved in order: `"dev"` → `"start"` → first script in package.json.

### 4.3 Python entry point detection

Look for entry files in order: `main.py` → `app.py` → `cli.py` → `__main__.py`. If none found but `pyproject.toml` has a `[project.scripts]` entry, use that.

### 4.4 Unknown stack

If no known stack markers are found, show a helpful error listing the detected files and suggesting the user add a `.projm.toml` override.

## 5. `.projm.toml` override

### 5.1 Location

Project root — `{project_root}/.projm.toml`

### 5.2 Schema

```toml
[run]
# Full command override — skips all detection for this project
command = "cargo run --features embedded"

# Named scripts shown in an interactive menu
# "default" is used when no selection flag is passed
scripts = [
    "dev",
    "build",
    "test",
]

[run.scripts]
# Explicit script commands (optional — otherwise inferred from stack)
dev   = "cargo run"
build = "cargo build --release"
test  = "cargo test"

[run.workspace_packages]
# Label the workspace package paths for the picker
api  = "packages/api"
web  = "apps/web"
core = "packages/core"
```

### 5.3 Evaluation order

```
1. .projm.toml [run] command          ← highest priority
2. .projm.toml [run.scripts] dev      ← named script override
3. Monorepo workspace detection        ← turbo/pnpm-workspace/nx
4. Stack auto-detection                ← built-in mapping
```

### 5.4 `.projm.toml` in parent directories

For monorepo roots, projm walks up from the project dir looking for `.projm.toml` in parent directories. If found at the monorepo root, the `[run.workspace_packages]` section configures package discovery.

## 6. Monorepo support

### 6.1 Detection

| Marker                    | Monorepo tool | Behaviour                                   |
| ------------------------- | ------------- | ------------------------------------------- |
| `turbo.json`              | Turborepo     | Read pipeline tasks, offer `turbo run dev`  |
| `pnpm-workspace.yaml`     | pnpm          | Read workspace globs, resolve packages      |
| `nx.json`                 | Nx            | Read project graph, offer `nx run <pkg>`    |
| `"workspaces"` in pkg.json | pnpm/yarn/bun | Resolve workspace package directories       |

### 6.2 Interactive picker for monorepos

When a monorepo is detected, `projm run` shows:

```
  Select target in drivetrack (monorepo):
  > Run All (dev)
    apps/web
    apps/api
    packages/shared
    packages/ui
```

- **"Run All"** executes the root dev command (e.g. `turbo run dev` or `pnpm -r dev`)
- **Individual packages** resolve to `turbo run dev --filter=<pkg>`, `pnpm --filter <pkg> dev`, etc.
- Escape/Ctrl+C exits without running anything.

### 6.3 Non-interactive monorepo

If a path or name is provided that resolves to a specific workspace package, skip the picker and run that package directly.

## 7. Implementation plan

### 7.1 New files

| File | Purpose |
|---|---|
| `src/run.rs` | `projm run` subcommand — detection, prediction, execution |
| `src/run_config.rs` | `.projm.toml` parsing (or inline in run.rs if small enough) |

### 7.2 Modified files

| File | Change |
|---|---|
| `src/main_cli.rs` | Add `Run` variant to `Commands` enum |
| `src/main.rs` | Add `Commands::Run => run::run()` dispatch |
| `src/lib.rs` | Add `pub mod run;` |
| `src/init_setup.rs` | Add `pn` shell function to shell profile block |
| `README.md` | Update roadmap: v0.7 ✓, add `pn` to usage |

### 7.3 Dependencies

No new external dependencies — `std::process::Command` is already used in `check.rs` and `go.rs`. `.projm.toml` uses the existing `toml` crate. The `go.rs` project listing logic will be extracted to `lib.rs` or a shared helper to avoid duplication.

### 7.4 Key design constraints

- **Ctrl+C must work** — the child process must be killed on signal. Use `Command::new(...).spawn()` and proxy the signal.
- **Streaming output** — use `spawn()` and pipe stdout/stderr line by line (or inherit fds), not `output()` which buffers.
- **UI on stderr** — all selection UI must go to stderr, stdout is reserved for dev command output, and the shell function handles the eval pattern.

### 7.5 Test plan

| Test | Description |
|---|---|
| Stack-to-command mapping | Unit test each marker → expected command |
| `.projm.toml` parsing | Valid/invalid config files |
| Monorepo detection | Turbo, pnpm, nx, bun workspace detection |
| Package.json script resolution | `dev` present, `dev` absent, `start` fallback, no scripts |
| Python entry detection | `main.py`, `app.py`, `pyproject.toml` scripts |
| `pn` shell function | Generated correctly in zsh and PowerShell |

## 8. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Dev command hangs or never exits | Ctrl+C forwards signal to child; `pn` is a foreground process |
| User stops `projm run` and CWD is now inside a child process | Shell handles this naturally — child dies with parent |
| Tauri / Flutter / Gradle have different project structures | Use same classification path from `classify.rs` — the single source of truth |
| `bun run dev` vs `bun dev` ambiguity | Check `package.json` scripts: if `dev` script exists → `bun run dev`, if not → `bun dev` (Bun's default) |
